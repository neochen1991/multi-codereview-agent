# 代码检视项目介绍（大白话版）

## 一句话先说清

这个工具就是一个“上线前质量把关员”。  
开发提代码后，它会先看改了什么，再把任务分给对应的“专家”，最后给出可落地的问题清单和修改建议，帮助团队把问题挡在上线前。

---

## 1. 本工具能做什么？

### 1.1 能帮团队解决什么痛点

1. 人工 Code Review 看得慢、看不全、标准不一致。  
2. 安全、性能、测试这类“跨专业问题”容易漏。  
3. 代码改了，和设计/规范有没有偏差，靠人盯很难。  
4. 问题发现晚，返工成本高。

### 1.2 现在能做到的事

1. 自动读取 MR/PR 的变更内容，按文件和代码段定位问题。  
2. 多个专家并行审查，不是单人视角。  
3. 每个问题都给证据（改动代码+关联上下文）和修复建议。  
4. 支持人工确认/驳回，形成闭环。  
5. 页面可看过程、看结论、看回放，便于复盘。

---

## 2. 实现方案（领导视角）

我们不是做一个“会聊天的机器人”，而是做一条“可控的质量流水线”。

### 2.1 流程怎么跑

1. 接收一次审查任务（MR/PR）。  
2. 解析变更文件和代码片段。  
3. 选专家并分工（用户手选就按用户选；没选才自动判断）。  
4. 各专家并行审查并输出结论。  
5. 系统合并结果，生成最终问题清单和报告。  
6. 人工可裁决，结果可回放。

### 2.2 方案的关键点

1. 只围绕“真实变更代码”做审查，不做全仓无差别扫描。  
2. 强制带上“代码上下文+规则规范”再审查，减少拍脑袋结论。  
3. 报告结构化输出，便于统计和治理。  
4. 前后端分层，后续可接企业平台（GitHub/GitLab/内部代码平台）。

---

## 3. 专家 Skill 是怎么实现的？

可以把它理解成：给专家配“工作手册 + 工具箱”。

### 3.1 工作手册

每个专家绑定自己的规范文档，明确三件事：

1. 该专家看什么。  
2. 不该专家看的边界是什么。  
3. 输出要包含哪些内容（证据、风险、建议）。

### 3.2 工具箱

Skill 会按条件触发工具，去补充专家需要的信息，比如：

1. 提取对应代码段。  
2. 查关联文件。  
3. 对照设计文档做一致性检查。

### 3.3 当前落地情况

当前已落地 1 个通用 Skill：`design-consistency-check`（设计一致性检查）。  
后续可继续按业务线扩展，不需要推翻整体架构。

---

## 4. 现在有多少专家？怎么划分？

### 4.1 总量

当前系统内置 **11 个专家**（已在仓库配置）。

### 4.2 按职责分层

#### A. 通用质量主线（默认常用 6 个）

1. `correctness_business`：业务正确性  
2. `architecture_design`：架构与设计  
3. `security_compliance`：安全与合规  
4. `performance_reliability`：性能与稳定性  
5. `maintainability_code_health`：可维护性与代码健康  
6. `test_verification`：测试充分性

#### B. 专项专家（按场景补充 5 个）

1. `database_analysis`：数据库与 SQL  
2. `redis_analysis`：缓存与 Redis  
3. `mq_analysis`：消息队列与异步链路  
4. `ddd_specification`：DDD 规范一致性  
5. `frontend_accessibility`：前端可访问性

### 4.3 这套划分的好处

1. 默认审查覆盖广，不漏核心质量项。  
2. 碰到专项风险可加专家，不靠“一个人全能”。  
3. 组织上容易落地，职责清晰，结果可追责。

---

## 5. 效果如何？

### 5.1 已经看到的效果

在现有 Java 真实场景回归里，系统已能稳定识别以下典型问题：

1. DDD 违规（绕过聚合工厂、事件顺序问题）。  
2. SQL/查询风险（模糊匹配放大、分页/limit 删除）。  
3. 稳定性问题（吞异常、批处理放大风险）。  
4. 命名和可维护性类问题（常量命名、代码健康项）。

### 5.2 管理层更关心的结果

1. 审查标准开始统一，不再完全依赖个人经验。  
2. 问题前移，减少“上线后再救火”。  
3. 审查过程可回放，便于团队复盘和治理。  
4. 平台化后可沉淀组织级规范资产。

### 5.3 当前客观状态（不回避）

1. 已具备上线试点能力。  
2. 还在持续优化阶段（速度、稳定性、误报率还在迭代）。  
3. 更适合“先试点、再规模化”。

---

## 6. 怎么推广（建议分三步）

### 第一步：小范围试点（2-4 周）

1. 选 1-2 个 Java 业务线先跑。  
2. 先不强制拦截，只做“旁路检视 + 人工确认”。  
3. 每周看三件事：检出率、误报率、处理时长。

### 第二步：扩面并建立规则共识（4-8 周）

1. 从 Java 扩到 JS/前端场景。  
2. 按团队共识沉淀“必查规则”。  
3. 把高价值问题接入发布门禁（先软门禁，再硬门禁）。

### 第三步：平台化运营（长期）

1. 建立月度质量看板（问题类型、重复问题、修复时效）。  
2. 形成“规范-检视-复盘-迭代”的固定机制。  
3. 把系统沉淀为组织级基础能力，对新团队可复制。

---

## 给领导的结论

这个项目不是“炫技工具”，而是“质量管理基础设施”。  
它的核心价值是：**把分散在个人经验里的审查能力，变成可执行、可复盘、可推广的组织能力**。  
建议路线：**先试点拿结果，再逐步推广到全团队。**
