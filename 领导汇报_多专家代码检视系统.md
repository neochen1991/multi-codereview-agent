## 一、本工具能做什么？

### 1.1 核心定位

**"AI驱动的代码质量守门员"** —— 在代码合并前自动发现潜在问题，降低生产故障率。

### 1.2 主要功能矩阵

| 功能模块 | 具体能力 | 业务价值 |
|---------|---------|---------|
| **智能代码审查** | 自动分析MR/PR代码变更，识别bug、安全漏洞、性能隐患 | 减少人工审查遗漏，提升代码质量 |
| **多维度专家会诊** | 6类专业专家（业务正确性、安全、性能等）协同审查 | 覆盖单一人工审查难以发现的跨领域问题 |
| **设计一致性校验** | 比对代码实现与详细设计文档的一致性 | 防止实现偏离设计意图 |
| **增量质量管控** | 基于Git Diff精准分析变更影响范围 | 避免全量扫描的资源浪费 |
| **人工闭环反馈** | 支持人工确认/驳回AI发现的问题，持续优化模型 | 形成质量改进闭环 |

### 1.3 典型应用场景

```
场景1：开发人员提交MR
    ↓
系统自动触发审查
    ↓
6个专家并行分析 → 发现4类问题
    ↓
审查结果推送 → 开发者修复 → 重新提交
    ↓
问题清零 → MR合并

场景2：架构师制定设计规范
    ↓
上传详细设计文档
    ↓
开发实现代码
    ↓
系统自动比对实现与设计的一致性
    ↓
发现偏离立即告警
```

## 三、专家Skill机制如何实现？

### 3.1 Skill机制概述

**Skill = 上层业务能力包**，负责定义"**什么时候触发、依赖哪些运行时工具、要求专家如何输出**"。

### 3.2 Skill实现架构

```
┌─────────────────────────────────────────────────────────────┐
│                    Skill加载与激活流程                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 系统启动时扫描 extensions/skills/ 目录                    │
│     └── 读取每个skill目录下的 SKILL.md + metadata.json        │
│                                                             │
│  2. 运行时判断Skill是否激活（多条件与逻辑）                     │
│     ├── 当前expert是否在bound_experts列表中？                   │
│     ├── 当前analysis_mode是否允许？                             │
│     ├── required_doc_types是否满足？                            │
│     ├── changed_files是否命中activation_hints？                 │
│     └── 必要上下文是否存在？                                    │
│                                                             │
│  3. 激活后的Skill执行流程                                       │
│     ├── 加载SKILL.md中声明的Required Tools                     │
│     ├── 调用ToolGateway执行工具                                  │
│     ├── 将工具结果注入专家Prompt                                 │
│     └── 专家根据SKILL.md中的Output Contract输出                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 Skill配置示例：设计一致性检查

**目录结构：**
```

├── SKILL.md          # Skill能力定义

```


### 3.4 Skill与Tool的关系

| 维度 | Skill | Tool |
|------|-------|------|
| **定位** | 上层业务能力包 | 底层执行插件 |
| **职责** | 定义"什么时候触发、依赖哪些工具、专家如何输出" | 负责真正执行检索、结构化提取和一致性比对 |
| **配置** | `SKILL.md` + `metadata.json` | `tool.json` + `run.py` |
| **绑定** | 绑定到Expert | 被Skill调用 |

## 四、有多少个专家，如何划分的？

### 4.1 专家总览

系统当前包含 **6个内置专家**，覆盖软件质量的核心维度。

### 4.2 专家详细配置

| 专家ID | 中文名 | 职责描述 | 适用文件模式 | 核心能力 |
|--------|--------|----------|--------------|----------|
| `correctness_business` | 业务正确性专家 | 检查业务逻辑、领域模型、业务流程正确性 | `**/service/**`, `**/domain/**`, `**/application/**` | 业务规则验证、领域模型一致性、设计文档对齐 |
| `correctness_technical` | 技术正确性专家 | 检查算法、数据结构、API使用正确性 | `**/*.java`, `**/*.py`, `**/*.ts` | 算法逻辑验证、API使用规范、技术实现最佳实践 |
| `security_compliance` | 安全合规专家 | 检查注入攻击、敏感数据、越权访问等安全风险 | `**/*Controller*`, `**/*Service*`, `**/api/**` | SQL注入检测、XSS防护、敏感数据脱敏、权限校验 |
| `performance_reliability` | 性能可靠专家 | 检查性能瓶颈、并发安全、资源泄漏 | `**/*Repository*`, `**/mapper/**`, `**/dao/**` | N+1查询检测、事务边界分析、并发安全问题、资源未释放 |
| `maintainability_code_health` | 可维护性专家 | 检查代码异味、复杂度、测试覆盖 | `**/*.java`, `**/*.py` | 圈复杂度分析、重复代码检测、命名规范、注释完整性 |
| `test_verification` | 测试验证专家 | 检查测试完整性、边界条件、Mock合理性 | `**/test/**`, `**/*Test*`, `**/*Spec*` | 测试用例覆盖、边界值测试、Mock使用规范、断言完整性 |

## 五、举个例子，什么专家发现了什么问题？

### 案例：电商订单状态变更代码审查

**背景：** 某开发者在订单服务中新增了一个取消订单功能，提交了以下代码变更：

```java
// OrderApplicationService.java (新增方法)
public void cancelOrder(Long orderId) {
    Order order = orderRepository.findById(orderId);
    // 检查订单状态
    if (order.getStatus().equals("PAID")) {
        order.setStatus("CANCELLED");
        orderRepository.save(order);
    }
}
```

### 各专家发现问题过程

| 专家 | 发现的问题 | 问题详情 | 风险等级 |
|------|-----------|---------|---------|
| **correctness_business<br>业务正确性专家** | 订单状态流转违规 | 只允许取消PAID状态订单，但未处理PAID之前的PENDING状态，且未检查库存释放、优惠券回退等业务规则 | High |
| **security_compliance<br>安全合规专家** | 越权访问风险 | 未验证当前用户是否有权取消该订单（如买家只能取消自己的订单） | Critical |
| **performance_reliability<br>性能可靠专家** | 事务边界问题 | `@Transactional`注解缺失，若库存回退和订单状态更新之间发生异常，会导致数据不一致 | High |
| **maintainability_code_health<br>可维护性专家** | 魔法字符串 | `"PAID"`、`"CANCELLED"`等状态使用硬编码字符串而非枚举，易因拼写错误导致bug | Medium |
| **test_verification<br>测试验证专家** | 测试覆盖不足 | 未提供边界条件测试（如已发货订单不可取消、并发取消场景） | Medium |

## 六、本工具的效果评估，采纳率平台？

### 6.1 当前评估指标

| 指标 | 说明 | 当前观测值 | 目标值 |
|------|------|-----------|--------|
| **检出率** | 发现的真实问题数 / 总问题数 | ~75% | ≥ 80% |
| **误报率** | 误报数 / 总报告数 | ~25% | ≤ 20% |
| **高置信度Issue占比** | 置信度≥0.8的Issue占比 | ~55% | ≥ 60% |
| **平均审查时间** | 单次MR审查耗时 | ~3-5分钟 | ≤ 5分钟 |
| **专家覆盖率** | 命中的专家数 / 总专家数 | ~60% | ≥ 70% |
| **规则命中率** | 命中的规则数 / 总规则数 | ~35% | ≥ 40% |

### 6.2 采纳率评估框架

**当前系统已具备的数据采集能力：**

```python
# 人工反馈数据模型 (FeedbackLabel)
class FeedbackLabel:
    review_id: str          # 审核ID
    issue_id: str           # 问题ID
    expert_id: str           # 专家ID
    feedback_type: str       # 反馈类型：confirm / reject / unclear
    severity_adjustment: int  # 严重度调整：-2 ~ +2
    comment: str             # 人工评语
    created_at: datetime     # 创建时间
```

**建议的采纳率计算方式：**

| 采纳率指标 | 计算公式 | 数据来源 |
|-----------|---------|---------|
| **Issue采纳率** | 被确认有效的Issue数 / 总Issue数 | `feedback_type=confirm` / 总Issue |
| **专家准确率** | 某专家的confirm数 / 该专家总发现数 | 按expert_id聚合统计 |
| **规则命中率** | 命中规则且被confirm数 / 总规则命中数 | `finding.matched_rules`关联统计 |
| **严重度调整率** | 被调整严重度的Issue占比 | `severity_adjustment != 0`的比例 |

## 七、能不能推广，工具具不具备推广条件？

### 7.1 推广可行性评估

| 评估维度 | 当前状态 | 是否具备推广条件 | 说明 |
|---------|---------|-----------------|------|
| **技术成熟度** | βeta阶段，核心功能稳定 | ⚠️ 基本具备 | 主要功能已跑通，但Windows内存问题需修复 |
| **接入成本** | 中等，需要配置Git仓库、LLM API | ⚠️ 需简化 | 一键部署脚本待完善 |
| **适用范围** | Java项目体验最佳 | ⚠️ 可扩展 | 当前聚焦Java，其他语言支持待开发 |
| **效果可量化** | 基础指标已有，深度分析待完善 | ⚠️ 需加强 | 需要更完善的采纳率看板 |
| **运维成本** | 单机部署，SQLite存储 | ✅ 具备 | 无需复杂运维，适合中小团队 |
| **定制化能力** | 专家、规则、Skill均可扩展 | ✅ 具备 | 插件化架构支持二次开发 |
