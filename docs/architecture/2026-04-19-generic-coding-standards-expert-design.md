# 将“架构与设计专家”重定义为“通用编码规范专家”设计稿

## 1. 背景

当前项目中的 `architecture_design` 专家，原始定位是：

- 看模块边界
- 看依赖方向
- 看抽象层级
- 看设计退化

这个定义本身没有问题，但在当前专家体系下会遇到两个明显问题：

### 问题 1：与其他专家边界重叠严重

当前它和以下专家有明显交集：

- `ddd_specification`
- `correctness_business`
- `maintainability_code_health`

真实结果就是：

- 同一类问题被多个专家同时提
- 尤其是“职责放错层”“实现不优雅”“设计退化”这类问题，表达容易泛
- 最终 merge 层只能被动去重，但源头没有真正收住

### 问题 2：当前项目更缺“通用编码规范检查”，而不是“泛架构评论”

在当前实际使用中，更常出现的问题是：

- 命名不规范
- 常量、枚举、异常使用不一致
- 空实现、魔法值、控制流不清晰
- 集合、判空、日志、异常处理方式不统一
- 注释风格和代码表达不一致

这类问题不是 DDD，也不是架构分层问题，但在 Java 项目里非常高频。

而且这些问题如果没有一个专门专家兜底，就会出现两种情况：

1. 被多个专家顺手提一下，重复很多  
2. 没人主提，结果漏掉

所以从当前体系演进来看，比较合理的方向不是“把架构能力删掉”，而是拆成两个更清楚的角色：

1. 把当前 `architecture_design` 收口成一个**通用编码规范专家**
2. 另外新增一个真正负责 **DDD + 架构边界** 的专项专家

这样既能补上当前最缺的“通用 Java 规范检查”，又不会把真正的架构视角丢掉。

---

## 2. 目标

将当前 `architecture_design` 专家重定义为：

**通用编码规范专家（Generic Coding Standards Expert）**

新的定位不是看宏观架构，而是：

- 基于阿里巴巴 Java 开发手册等通用 Java 编码规范
- 检查常规、高频、跨项目普适的编码问题
- 统一承接“非业务语义、非专项中间件、非领域建模”的通用代码质量问题

一句话说：

**它负责的是“普通 Java 代码写得是不是规整、稳妥、可长期维护”，而不是“系统架构是不是合理”。**

同时，补充新增一个新专家：

**DDD 架构专家（DDD Architecture Expert）**

这个新专家负责：

- 聚合边界
- 上下文边界
- 分层职责
- 依赖方向
- 应用层 / 领域层 / 基础设施层的职责归属

一句话说：

**通用编码规范专家看“代码写法”，DDD 架构专家看“职责和边界”。**

---

## 3. 非目标

这次调整不打算让“通用编码规范专家”继续承担以下职责：

1. 不再主提通用分层与依赖方向问题  
   这类问题后续应由新增的 **DDD 架构专家** 承担。

2. 不代替业务专家判断业务规则是否正确  
   行为是否错，仍由 `correctness_business` 主提。

3. 不代替数据库、性能、安全等专项专家  
   事务、锁、索引、并发、安全边界仍交给对应专家。

4. 不做格式化工具级别的问题扫描  
   比如单纯缩进、空格、import 排序，不应该成为主要 finding 来源。

## 4. 目标专家结构

### 4.1 通用编码规范专家

沿用现有 expert id：

- 专家 ID：**保留** `architecture_design`
- 英文展示名：`generic-coding-standards`
- 中文展示名：`通用编码规范专家`

### 4.2 新增 DDD 架构专家

建议新增一个新 expert，而不是继续让 `ddd_specification` 和旧 `architecture_design` 分担。

建议：

- 专家 ID：`ddd_architecture`
- 英文展示名：`ddd-architecture`
- 中文展示名：`DDD 架构专家`

这个新专家专门承接：

- 分层边界
- 依赖方向
- 聚合边界
- 应用服务与领域服务职责
- 上下文边界
- 基础设施实现是否向上泄漏

这样角色会比当前清楚很多：

- **通用编码规范专家**：看“代码写法”
- **DDD 架构专家**：看“职责和边界”
- **可维护性专家**：看“长期演化成本”

## 5. 通用编码规范专家定位
### 新角色描述

> 关注 Java 项目中的通用编码规范、常规实现质量和工程一致性，基于阿里巴巴 Java 开发手册及团队通用编码约定，识别高频、普适、非专项的代码问题。

---

## 6. 规范依据

建议该专家主要参考以下规范体系：

### 5.1 主参考

- 《阿里巴巴 Java 开发手册》

建议重点采用其中对 Code Review 最有价值的部分，而不是全量照搬：

- 命名规范
- 常量与枚举使用
- 集合与泛型使用
- 异常处理规范
- 日志规范
- 控制流与判空
- 并发工具基础用法
- 面向对象设计中的基础实现习惯

### 5.2 辅助参考

- Google Java Style Guide（仅作为补充）
- 团队内部 Java 代码约定

---

## 7. 通用编码规范专家以后主要看什么

下面这张表是建议的主责范围。

| 类别 | 是否主责 | 说明 |
|---|---|---|
| 命名规范 | 是 | 类名、方法名、变量名、布尔命名、常量命名等 |
| 常量与枚举使用 | 是 | 魔法值、状态值硬编码、字符串常量滥用 |
| 判空与默认值处理 | 是 | 不必要的空指针风险、判空顺序混乱、默认值写法不稳 |
| 异常处理方式 | 是 | 吞异常、异常粒度不合适、异常语义不清、catch 后无动作 |
| 日志规范 | 是 | 日志缺失、日志级别不合适、打印无关信息、丢失关键信息 |
| 集合与泛型使用 | 是 | 原始类型、无意义拷贝、集合遍历和删除写法危险 |
| 控制流表达 | 是 | 条件分支混乱、早返回缺失、嵌套太深、代码意图不清 |
| 基础面向对象实现习惯 | 是 | equals/hashCode/toString 使用不当，职责落在错误对象上但还未到架构级问题 |
| 基础并发使用习惯 | 是，但仅限常规写法 | 线程池、锁、并发容器的常规误用；不承担系统级性能问题 |
| 可读性和一致性 | 是 | 同类代码风格不一致、团队通用写法偏差明显 |

## 8. 通用编码规范专家典型会提什么问题

### 7.1 命名问题

- 布尔变量命名不表达语义
- 方法名像实现步骤，不像业务动作
- DTO / VO / Entity / Command 命名混乱

### 7.2 魔法值问题

- 业务状态直接写 `"1"`、`"2"`、`"SUCCESS"`
- 超时时间、重试次数、分页大小直接写死

### 7.3 空值与控制流问题

- 先取值再判空
- 多层嵌套 if/else，读不清主路径
- 异常路径和正常路径混在一起

### 7.4 异常处理问题

- `catch (Exception e)` 后只打日志，不做收口
- 把异常吞掉，调用方无法感知
- 抛出的异常语义和实际问题不匹配

### 7.5 日志问题

- 关键失败路径没有日志
- 打印了无意义日志
- 关键上下文缺失，后续无法定位问题

### 7.6 集合与对象使用问题

- 原始类型集合未加泛型
- 对象拷贝与集合转换逻辑混乱
- 循环中做危险修改

### 7.7 一致性问题

- 同模块类似逻辑风格差异很大
- 同一类对象在多个地方用不同约定

## 9. 通用编码规范专家明确不应该看什么

为了防止重复，这个专家必须明确**不主提**以下问题：

### 不看业务语义正确性

这些交给 `correctness_business`：

- 状态流转是否正确
- 业务规则是否被改坏
- 注释 / 接口承诺是否真正实现

### 不看 DDD 领域边界和架构边界

这些改交给新增的 `ddd_architecture`：

- 聚合边界被绕过
- 应用服务堆业务逻辑
- 领域对象职责丢失
- 上下文边界和依赖方向错误
- 跨层直连和基础设施实现泄漏

### 不看数据库专项问题

这些交给 `database_analysis`：

- SQL 语义
- 事务边界
- schema 演进
- 索引、锁、约束

### 不看系统级性能与稳定性

这些交给 `performance_reliability`：

- 热点路径退化
- 批处理过大
- 锁竞争放大
- 超时 / 重试 / 回滚 / 降级

### 不看安全专项

这些交给 `security_compliance`：

- 鉴权授权
- 输入校验绕过
- 敏感数据风险

### 不看测试覆盖问题

这些交给 `test_verification`：

- 缺测试
- 断言薄弱
- 回归保护不足

## 10. DDD 架构专家的职责设计

这个新专家的定位要尽量硬，避免再次变成泛泛的“设计专家”。

### 10.1 核心职责

- 看分层边界是不是被破坏
- 看依赖方向是不是反了
- 看聚合边界是不是被绕过
- 看应用服务是否承担了不该承担的业务规则
- 看基础设施实现是否向上层泄漏

### 10.2 主提范围

- 聚合边界被跨层绕过
- 应用服务堆业务逻辑
- 领域服务职责错位
- 上下文边界不清
- 依赖方向错误
- 跨层直连
- 抽象层级塌陷
- 基础设施细节上浮

### 10.3 不主提的范围

- 命名、魔法值、判空、异常、日志等通用编码规范问题  
  -> 交给通用编码规范专家
- 业务行为对不对  
  -> 交给 `correctness_business`
- SQL、事务、索引  
  -> 交给 `database_analysis`
- 系统级并发、超时、重试、资源效率  
  -> 交给 `performance_reliability`
- 长期维护性和重构切分点  
  -> 交给 `maintainability_code_health`

### 10.4 一句话

它负责回答：

**这段代码的职责和边界有没有放错。**

## 11. 与现有专家的边界调整建议

### 11.1 与 `maintainability_code_health`

这是最容易重叠的一组。

建议拆法：

- **通用编码规范专家**  
  看“写法是否符合 Java 通用规范”
- **可维护性专家**  
  看“这段代码长期好不好维护”

也就是说：

- 命名、常量、判空、异常、日志、一致性 -> 通用编码规范专家
- 职责揉在一起、函数过长、抽象不清、演化成本高 -> 可维护性专家

### 11.2 与 `correctness_business`

建议拆法：

- 注释 / 接口 / TODO 承诺未实现 -> `correctness_business`
- 注释风格不清、名字表达差、代码读起来别扭 -> 通用编码规范专家

### 11.3 与新增 `ddd_architecture`

建议硬拆成这样：

- **通用编码规范专家**
  - 看写法、规范、一致性
- **DDD 架构专家**
  - 看职责、边界、依赖方向

一句话区分：

- 前者回答“写得规不规范”
- 后者回答“职责放得对不对”

## 11.4 关键专家职责矩阵

下面这张表专门用来区分当前最容易重叠的四个专家：

- 通用编码规范专家
- DDD 架构专家
- 正确性与业务专家
- 可维护性与代码健康专家

这四个专家的差异如果不说清楚，后续还是会重复提问题。

| 维度 | 通用编码规范专家 | DDD 架构专家 | 正确性与业务专家 | 可维护性与代码健康专家 |
|---|---|---|---|---|
| 核心问题 | 写法规不规范 | 职责和边界对不对 | 行为和结果对不对 | 后续好不好维护 |
| 主要关注 | 命名、常量、判空、异常、日志、集合、控制流、一致性 | 聚合边界、依赖方向、分层职责、上下文边界、基础设施泄漏 | 业务规则、状态流转、边界条件、输入输出一致性、承诺未实现 | 函数复杂度、职责揉在一起、重复代码、抽象质量、演化成本 |
| 典型问题 | 魔法值、布尔命名差、catch 后吞异常、日志缺关键信息 | 应用服务堆业务规则、聚合被绕过、跨层直连、依赖方向反了 | 状态流转错、边界输入没处理、注释说了但代码没实现、返回值和副作用不一致 | 函数过长、分支太深、复制粘贴、结构难懂、后续修改成本高 |
| 看到了但不要主提 | 业务行为错、聚合边界错、索引和事务问题、性能瓶颈 | 命名和日志问题、纯风格问题、SQL 细节、系统级性能问题 | 命名规范、可读性、架构优雅性、索引设计 | 业务规则对错、分层职责对错、性能和安全专项问题 |
| 最容易混淆的对象 | `maintainability_code_health` | `ddd_specification` / 旧 `architecture_design` | `ddd_architecture` / `security_compliance` | 通用编码规范专家 |
| 最终归因原则 | “这是写法问题”就归这里 | “这是边界和职责问题”就归这里 | “这是行为错误”就归这里 | “这是长期维护成本问题”就归这里 |

### 怎么用这张表

后续如果一个问题同时满足多个角度，优先按下面这个顺序判断最终归因：

1. **先看是不是行为错误**  
   如果会直接导致业务结果错、状态错、承诺没实现，先归 `correctness_business`

2. **再看是不是职责和边界错误**  
   如果核心是跨层、聚合边界、依赖方向或职责放错层，归 `ddd_architecture`

3. **再看是不是通用编码规范问题**  
   如果核心是命名、常量、异常、日志、判空、控制流写法不规范，归通用编码规范专家

4. **最后看是不是长期维护成本问题**  
   如果代码行为没错、边界也没错，但结构让后续演进明显变难，归 `maintainability_code_health`

### 两个典型例子

#### 例子 1：注释说“失败自动降级”，代码里没有降级逻辑

- 不归通用编码规范专家
- 不归可维护性专家
- 归 `correctness_business`

原因：

- 这是行为承诺没有落地
- 会直接影响结果正确性

#### 例子 2：应用服务里直接 new 基础设施实现，还顺手写了一堆流程编排

- 不归通用编码规范专家
- 不优先归可维护性专家
- 归 `ddd_architecture`

原因：

- 核心不是“写法不好”
- 核心是职责和边界被放错了

#### 例子 3：方法名模糊、状态值用字符串硬编码、日志里缺关键主键

- 归通用编码规范专家

原因：

- 这是典型的 Java 通用编码规范问题
- 不需要上升到架构或业务语义层

#### 例子 4：代码能跑，但一个函数塞了十几个分支和多段重复逻辑

- 归 `maintainability_code_health`

原因：

- 主要风险是后续维护和演化成本
- 不是行为错误，也不是分层边界错误

## 12. Prompt 层建议

后续如果真正落代码，prompt 应体现以下几条：

### 12.1 通用编码规范专家的 prompt

- 你负责检查 Java 项目中的通用编码规范问题
- 你关注的是实现质量和工程一致性，不是业务语义和专项架构问题

### 12.2 DDD 架构专家的 prompt

应体现：

- 你负责检查聚合边界、上下文边界、分层职责和依赖方向
- 你不负责命名、常量、日志、判空、异常等通用编码规范问题
- 你必须指出“哪一层依赖了哪一层”“哪个职责放错了”“边界是如何被绕过的”

### 12.3 输出原则

- 只输出高价值、会影响维护性、稳定性或团队一致性的问题
- 不要输出纯格式化层面的低价值提醒
- 结论必须有具体代码证据

### 12.4 边界约束

- 业务逻辑是否正确 -> 交给 `correctness_business`
- DDD 边界 -> 交给 `ddd_specification`
- 数据库语义 -> 交给 `database_analysis`
- 性能瓶颈 -> 交给 `performance_reliability`

## 13. 路由层建议

后续如果落代码，建议分别给两个专家不同的触发信号。

### 13.1 通用编码规范专家的触发信号

例如：

- util
- helper
- constant
- enum
- exception
- logger
- dto
- vo
- converter
- mapper
- validator

同时降低以下信号权重：

- orchestrator
- application
- runtime
- module

因为这些更偏旧的架构专家角色。

### 13.2 DDD 架构专家的触发信号

例如：

- domain
- aggregate
- entity
- valueobject
- application
- service
- repository
- orchestrator
- module
- boundedcontext
- domainservice

## 14. 收益

这样改之后，系统能得到几个实际收益：

### 收益 1：把“写法问题”和“边界问题”真正拆开

当前最模糊的一类问题就是：

- 到底是代码写得不规范
- 还是职责边界有问题

拆成两个专家之后，这类问题会清楚很多。

### 收益 2：减少与 DDD / 正确性 / 可维护性专家的重叠

当前最模糊的“泛设计问题”会明显减少。

### 收益 3：补上 Java 团队高频但没人主责的问题

比如：

- 命名规范
- 魔法值
- 判空与异常处理
- 日志与常量规范

这些问题在 Java 项目里很常见，但现在没有一个职责明确的专家兜底。

### 收益 4：让专家体系更贴实际开发场景

当前项目里，真正高频的问题很多不是架构级，而是“普通 Java 代码写得不够规范、风险埋得很浅但很多”。

把这个专家调成通用编码规范专家，更符合实际价值。

---

## 15. 风险

### 风险 1：如果通用编码规范专家收口过宽，会和 `maintainability_code_health` 再次重叠

所以必须把边界说死：

- 通用编码规范 -> 常规 Java 规范与一致性
- 可维护性 -> 长期演化成本

### 风险 2：如果 DDD 架构专家边界不够硬，会再次和 `correctness_business`、`ddd_specification` 重叠

所以必须明确：

- 它只看职责和边界
- 不看业务行为对错
- 不看通用编码规范

### 风险 3：如果照搬阿里规范过多，容易报出大量低价值问题

所以必须明确：

- 不做纯格式化检查
- 不做机械式“手册逐条背诵”
- 只保留对代码质量真正有影响的问题

### 风险 4：短期内会影响现有测试和回归口径

因为当前系统里很多地方都假定 `architecture_design` 还是架构专家。  
所以这次先做设计文档，不直接改代码，是合理的。

---

## 16. 分阶段实施建议

### 阶段 1：只确定设计和边界

输出这份设计文档，不改代码。

### 阶段 2：改两个专家定义

后续再改：

- `architecture_design` -> 通用编码规范专家
- 新增 `ddd_architecture`
- 各自的 `expert.yaml`
- 各自的 `prompt.md`
- 各自的导出规范文档

### 阶段 3：改 merge 归因规则

确保：

- 命名 / 常量 / 判空 / 异常 / 日志 / 一致性问题
  -> 只由通用编码规范专家主提
- 聚合边界 / 依赖方向 / 分层职责 / 上下文边界问题
  -> 只由 DDD 架构专家主提

### 阶段 4：跑真实 MR 验证

验证两件事：

1. 常规 Java 规范问题是否更容易被稳定提出来  
2. 架构和 DDD 边界问题是否不再四处飘  
3. 与其他专家的重复问题是否明显下降

---

## 17. 结论

当前最合理的方案不是“把架构专家改名”这么简单，而是：

1. 把当前 `architecture_design` 收口成  
   **基于阿里巴巴 Java 开发手册的通用编码规范专家**
2. 同时新增一个  
   **DDD 架构专家**

这样最终形成的结构是：

- 通用编码规范专家：看代码写法
- DDD 架构专家：看职责和边界
- 可维护性专家：看长期演化成本

一句话总结：

**这次不是减少专家，而是把一个边界模糊的角色拆成两个职责明确的角色。**

---

## 18. 可直接落到 prompt 的职责边界文案

这一节的目标不是继续讲设计，而是给后续代码改造提供可直接复用的 prompt 文案。

建议后续每个专家 prompt 都统一增加两部分：

1. **你负责什么**
2. **你不要主提什么，应该交给谁**

下面先给出四个最关键专家的建议文案。

### 18.1 通用编码规范专家

#### 建议的角色描述

你是通用编码规范专家。  
你的任务是基于阿里巴巴 Java 开发手册和团队通用 Java 编码约定，识别当前改动中的常规编码规范问题和工程一致性问题。

#### 建议的“你负责什么”

- 识别命名不规范、常量和枚举使用不当、魔法值硬编码的问题
- 识别判空、默认值、异常处理、日志使用、集合与泛型使用上的常规风险
- 识别控制流表达混乱、代码风格不一致、工程实现习惯不统一的问题
- 只输出会影响可读性、稳定性或团队一致性的高价值编码规范问题

#### 建议的“你不要主提什么”

- 不要代替正确性与业务专家判断业务规则、状态流转和行为正确性
- 不要代替 DDD 架构专家判断聚合边界、分层职责和依赖方向
- 不要代替数据库专家判断 SQL、事务、索引和 schema 问题
- 不要代替性能专家判断并发、锁、超时和系统级性能风险
- 不要代替可维护性专家判断长期演化成本和重构切分点

#### 建议的归因原则

如果问题的核心是“这段代码写法不规范、表达不稳妥、团队一致性不好”，归你。  
如果问题的核心是“行为错了”或“职责放错了”，不要主提。

---

### 18.2 DDD 架构专家

#### 建议的角色描述

你是 DDD 架构专家。  
你的任务是识别当前改动是否破坏了聚合边界、上下文边界、分层职责和依赖方向。

#### 建议的“你负责什么”

- 识别聚合边界被绕过、领域规则跑出领域层的问题
- 识别应用服务堆业务逻辑、领域对象职责丢失的问题
- 识别跨层直连、依赖方向错误、基础设施实现细节向上泄漏的问题
- 识别上下文边界不清、模块职责错位和抽象层级退化的问题

#### 建议的“你不要主提什么”

- 不要代替通用编码规范专家评论命名、日志、魔法值、判空和异常写法
- 不要代替正确性与业务专家判断业务行为是否错误
- 不要代替数据库专家判断 SQL、事务和索引问题
- 不要代替性能专家判断系统级性能和失败恢复问题
- 不要代替可维护性专家评论一般性的重构建议或复杂度问题

#### 建议的归因原则

如果问题的核心是“职责放错层了”“边界被绕过了”“依赖方向不对”，归你。  
如果只是写法不规范或逻辑行为错误，不要主提。

---

### 18.3 正确性与业务专家

#### 建议的角色描述

你是正确性与业务专家。  
你的任务是判断当前改动是否会让业务行为、状态流转、输入输出或需求语义发生错误。

#### 建议的“你负责什么”

- 识别业务规则被改坏、状态流转错误、边界条件漏处理的问题
- 识别输入输出和副作用不一致的问题
- 重点核对注释、方法名、接口说明、TODO 承诺的行为是否真的落地
- 输出时必须说清楚：什么情况下会错、错在哪里、会影响谁

#### 建议的“你不要主提什么”

- 不要代替通用编码规范专家评论命名、常量、日志、判空和风格一致性
- 不要代替 DDD 架构专家判断职责边界和分层归属
- 不要代替数据库专家评价索引和事务细节
- 不要代替性能专家预测容量和压测级性能问题
- 不要代替测试专家判断测试覆盖率是否充分

#### 建议的归因原则

如果问题的核心是“行为会错”“承诺没实现”“状态会跑偏”，归你。  
如果核心不是行为错误，而是写法或边界问题，就不要主提。

---

### 18.4 可维护性与代码健康专家

#### 建议的角色描述

你是可维护性与代码健康专家。  
你的任务是识别那些虽然当前能跑，但已经明显提高后续维护、调试、测试和演化成本的实现。

#### 建议的“你负责什么”

- 识别函数过长、职责揉在一起、分支过深、重复代码过多的问题
- 识别抽象不清、结构难懂、后续修改风险高的问题
- 识别会显著抬高测试和调试成本的实现方式
- 优先指出长期演化成本，而不是短期功能是否能跑

#### 建议的“你不要主提什么”

- 不要代替通用编码规范专家评论命名、日志、常量、判空等常规规范问题
- 不要代替正确性与业务专家判断业务语义是否错误
- 不要代替 DDD 架构专家判断职责边界和依赖方向
- 不要代替性能专家预测系统级性能瓶颈
- 不要代替安全专家判断安全漏洞是否成立

#### 建议的归因原则

如果问题的核心是“后面改起来会越来越难”，归你。  
如果问题的核心是“这段代码现在就会跑错”或“边界已经放错层”，不要主提。

---

## 19. Prompt 改造落地顺序建议

为了降低风险，建议按下面顺序落：

### 第一步：先只改边界文案

先不改专家路由，不改 merge，只把四个专家 prompt 的职责边界写清楚：

- 通用编码规范专家
- DDD 架构专家
- 正确性与业务专家
- 可维护性与代码健康专家

这样可以先观察：

- 重复问题有没有下降
- 是否出现“没人提”的空白位

### 第二步：再改主责归因

等 prompt 边界稳定后，再改 merge 归因：

- 写法问题 -> 通用编码规范专家
- 职责边界问题 -> DDD 架构专家
- 行为错误 -> 正确性与业务专家
- 演化成本问题 -> 可维护性专家

### 第三步：最后再调专家路由

如果前两步效果稳定，再去调 activation hints 和主 Agent 路由，避免一上来改太多，难以判断是哪一步起作用。

---

## 20. 专家路由与归因规则草案

这一节回答两个问题：

1. 主 Agent 什么时候应该优先选择哪个专家
2. 多个专家都提了时，最终结果归谁

---

### 20.1 路由原则

路由不要再只靠文件名或目录名，而是尽量靠**问题信号**。

建议分成四类信号：

#### A. 通用编码规范信号

当代码里出现这些信号时，优先考虑通用编码规范专家：

- 魔法值、硬编码状态值
- 布尔命名不清
- catch 后吞异常
- 日志缺关键字段
- 常量 / 枚举使用混乱
- 判空顺序危险
- 集合和泛型使用不规范
- 代码风格明显不一致

#### B. DDD 架构信号

当代码里出现这些信号时，优先考虑 DDD 架构专家：

- application/service/repository/domain 之间职责交叉
- 聚合、实体、值对象、领域服务相关改动
- 应用服务承担业务规则
- 基础设施实现上浮到上层
- 跨模块直接依赖
- bounded context、aggregate、domain 相关命名

#### C. 正确性信号

当代码里出现这些信号时，优先考虑正确性与业务专家：

- comment / 注释 / TODO / FIXME / 未实现
- 状态流转条件
- 返回值与持久化结果关系
- 边界输入、空值、异常分支
- 业务流程关键判断

#### D. 可维护性信号

当代码里出现这些信号时，优先考虑可维护性专家：

- 超长函数
- 深层嵌套
- 多段复制粘贴
- 结构复杂但行为没明显错
- 一个对象承担太多职责

---

### 20.2 路由优先级

当多个信号同时存在时，建议按下面顺序选择“主问题视角”：

1. **行为会不会错**  
   -> `correctness_business`

2. **职责和边界有没有错位**  
   -> `ddd_architecture`

3. **是不是常规编码规范问题**  
   -> 通用编码规范专家

4. **是不是长期维护成本问题**  
   -> `maintainability_code_health`

一句话说：

**行为优先于边界，边界优先于写法，写法优先于维护成本。**

---

### 20.3 merge 归因规则

后续 merge 层如果遇到多个专家都提了，建议按下面规则收口。

#### 规则 1：行为错误优先

如果一条问题同时满足：

- 注释没实现
- 命名也不好
- 结构也一般

最终优先归到：

- `correctness_business`

因为最核心的问题是：**行为没落地**。

#### 规则 2：边界错误优先于写法问题

如果一段代码同时存在：

- 依赖方向错误
- 命名不清

最终优先归到：

- `ddd_architecture`

因为最核心的问题是：**职责和边界错了**。

#### 规则 3：写法问题优先于纯维护建议

如果一段代码没有行为错误，也没有边界错误，但：

- 魔法值很多
- 日志不规范
- 判空写法危险

最终优先归到：

- 通用编码规范专家

只有当它不属于具体规范问题，而是“整体结构长期不易维护”，才归可维护性专家。

---

## 21. 典型问题归因示例

下面给几组更贴近真实 MR 的归因示例。

| 场景 | 应归专家 | 原因 |
|---|---|---|
| 注释写着“失败自动降级”，代码里没有降级逻辑 | `correctness_business` | 行为承诺未落地 |
| service 里直接 new repository implementation 并夹带业务规则 | `ddd_architecture` | 职责和边界错位 |
| 方法里大量硬编码字符串状态值 | 通用编码规范专家 | 典型常规 Java 规范问题 |
| 一个函数 200 行，分支很多，复制粘贴多，但行为暂时没错 | `maintainability_code_health` | 主要风险是长期演化成本 |
| catch 后只打日志不处理，且日志里没有关键主键 | 通用编码规范专家 | 异常与日志使用不规范 |
| 输入校验没做导致恶意参数可绕过校验 | `security_compliance` | 核心是安全边界，不是业务边界 |
| SQL 删除了 LIMIT，批量读取边界消失 | `database_analysis` | 核心是数据库语义和数据边界 |
| 批处理过大导致锁持有时间过长，系统压力被放大 | `performance_reliability` | 核心是系统级稳定性和故障放大 |

## 22. 这份设计文档后续怎么用

这份文档后面建议按三种方式使用：

### 用法 1：改 prompt

把第 18 节里的边界文案直接贴进对应专家 prompt。

### 用法 2：改专家职责说明

把第 11 节和第 20 节里的主责范围，写进：

- `expert.yaml`
- 专家中心展示文案
- 对外介绍材料

### 用法 3：改 merge / judge 逻辑

把第 20、21 节变成程序里的归因规则，避免结果页重复。

## 23. 当前建议的最终结构

如果按这份设计走，最终最关键的四个专家会变成：

1. **通用编码规范专家**  
   看普通 Java 代码写法是否规范

2. **DDD 架构专家**  
   看职责、边界和依赖方向是否正确

3. **正确性与业务专家**  
   看行为和结果是否正确

4. **可维护性与代码健康专家**  
   看后续维护和演化成本

这四个角色一旦边界清楚，当前最常见的重复问题基本就能压下去。
