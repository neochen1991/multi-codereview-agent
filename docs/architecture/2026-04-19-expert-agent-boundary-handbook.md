# 专家 Agent 职责边界手册

## 1. 文档目的

这份文档的目标只有一个：

**把每个专家 Agent 的职责边界讲清楚，避免多个专家对同一类问题重复发声。**

当前系统的问题不是专家数量不够，而是几个关键专家之间边界不够硬：

- 同一类问题会被多个专家同时提出
- 结果页出现“换个说法重复提”的现象
- merge 虽然会做去重，但源头没有真正收口

尤其是下面几类问题最容易重叠：

- 命名规范、可读性、魔法值
- 注释 / TODO / 接口承诺未实现
- 架构分层与 DDD 职责
- 数据库语义问题与性能稳定性问题

所以这份文档不是在介绍专家，而是在定义：

1. 每个专家**必须负责什么**
2. 每个专家**不要主提什么**
3. 同一类问题最终**应该归谁**

---

## 2. 总体设计原则

### 原则 1：一个问题类别，只允许一个主责专家

例如：

- 命名规范 -> 只能由通用编码规范专家主提
- 注释 / TODO / 接口承诺未实现 -> 只能由正确性与业务专家主提
- 聚合边界被绕过 -> 只能由 DDD 架构专家主提

其他专家即使看到了，也只能作为背景判断，不应该主提。

### 原则 2：专家按“问题责任边界”划分，不按技术名词堆角色

例如：

- 通用编码规范专家负责“写法是不是规整”
- DDD 架构专家负责“职责和边界是不是放对”
- 正确性与业务专家负责“行为是不是会错”
- 可维护性专家负责“后面改起来会不会越来越难”

### 原则 3：merge 层只保留唯一主责归因

如果多个专家都发现了同一个问题，结果层必须只保留一条，并归到唯一主责专家。

### 原则 4：行为优先于边界，边界优先于写法，写法优先于维护成本

也就是：

1. 先看是不是行为错误
2. 再看是不是职责和边界错误
3. 再看是不是通用编码规范问题
4. 最后看是不是长期维护成本问题

---

## 3. 建议的专家分层结构

为了解决当前重叠问题，建议把专家体系理解成两层：

### 3.1 通用层专家

这层负责高频、普适、跨项目共通的问题。

1. 通用编码规范专家
2. DDD 架构专家
3. 正确性与业务专家
4. 可维护性与代码健康专家
5. 测试与验证专家
6. 安全与合规专家

### 3.2 专项层专家

这层负责中间件、基础设施和专项场景。

1. 数据库分析专家
2. Redis 分析专家
3. MQ 分析专家
4. 性能与可靠性专家
5. 前端可访问性专家

### 3.3 当前与目标结构说明

当前系统内置专家是 11 个，主要包括：

- `correctness_business`
- `architecture_design`
- `ddd_specification`
- `maintainability_code_health`
- `database_analysis`
- `performance_reliability`
- `redis_analysis`
- `mq_analysis`
- `security_compliance`
- `test_verification`
- `frontend_accessibility`

目标结构建议做两处收口：

1. 把 `architecture_design` 收口成 **通用编码规范专家**
2. 新增一个真正的 **DDD 架构专家** `ddd_architecture`

这样原来模糊的“架构与设计”角色就被拆成两个明确职责：

- 看代码写法 -> 通用编码规范专家
- 看职责和边界 -> DDD 架构专家

| MQ 分析专家 | 消息顺序、幂等、ack、重试、补偿、死信、堆积治理 | 重复消费、顺序错乱、死信处理缺失、消息丢失风险、堆积治理缺口 | 不看 SQL 和索引，不评论前端问题，不主提通用代码风格问题 |
| 性能与可靠性专家 | 热点路径、并发稳定性、资源效率、超时重试、失败恢复 | 批处理过大、锁竞争放大、同步阻塞、资源泄漏、超时重试缺口、局部故障放大成系统压力 | 不主提索引设计和 SQL 字段问题，不主提命名和风格问题 |
| 安全与合规专家 | 鉴权授权、输入校验、敏感数据、合规边界 | 权限绕过、输入校验绕过、敏感信息泄露、合规风险 | 不主提一般边界条件问题，不判断普通性能瓶颈，不评论可读性 |
| 测试与验证专家 | 自动化测试覆盖、断言质量、回归保护、验证步骤 | 缺测试、断言太弱、风险路径无保护、缺集成测试、缺回归脚本或人工验证清单 | 不判断业务规则本身是否正确，不评论命名风格，不主提架构边界问题 |
| 前端可访问性专家 | 前端 a11y、关键交互可达性、基础渲染可达性 | a11y 回归、关键交互不可达、基础渲染或可达性退化 | 不看后端业务逻辑，不看数据库、中间件，不评论通用 Java 编码规范 |

### 这张表怎么用

研发同学可以按下面这个顺序理解：

1. 先问：这个问题是**行为错了**，还是**边界放错了**，还是**写法不规范**  
2. 再看它最接近哪一行专家的职责范围  
3. 如果一个问题能被多个专家“看见”，以表格里的“主要职责范围”为准，只保留一个主责专家

一句话总结：

**同一个问题可以被多个专家看见，但最终只能由一个主责专家提出。**

## 6. 关键专家边界定义

下面这一节只讲最核心的职责边界，不讲实现细节。

### 6.1 通用编码规范专家

#### 核心职责

基于阿里巴巴 Java 开发手册和团队通用 Java 约定，识别常规、高频、普适的编码规范问题和工程一致性问题。

#### 主提范围

- 命名不规范
- 魔法值和硬编码状态值
- 判空与默认值写法不稳
- 异常处理方式不合理
- 日志缺失或语义不足
- 集合与泛型使用不规范
- 控制流表达不清
- 同模块编码习惯明显不一致

#### 不要主提

- 业务规则是否错误
- 注释或接口承诺是否没实现
- 聚合边界与分层职责问题
- SQL / 事务 / 索引
- 系统级性能与锁问题
- 测试覆盖不足

#### 一句话

它负责回答：

**这段 Java 代码写得规不规范。**

---

### 6.2 DDD 架构专家

#### 核心职责

识别聚合边界、上下文边界、分层职责和依赖方向是否被破坏。

#### 主提范围

- 聚合边界被绕过
- 应用服务堆业务逻辑
- 领域对象职责丢失
- 上下文边界不清
- 分层职责错位
- 依赖方向错误
- 跨层直连
- 基础设施实现上浮

#### 不要主提

- 命名、日志、魔法值、判空等通用规范问题
- 业务行为对不对
- SQL、事务、索引细节
- 系统级性能和故障恢复
- 一般性的重构建议

#### 一句话

它负责回答：

**这段代码的职责和边界放得对不对。**

---

### 6.3 正确性与业务专家

#### 核心职责

判断当前改动是否会让业务行为、状态流转、输入输出或需求语义发生错误。

#### 主提范围

- 业务规则错误
- 状态流转错误
- 输入输出与副作用不一致
- 边界条件漏处理
- 异常分支导致行为跑偏
- 注释、方法名、接口说明、TODO 承诺的行为没有真正实现

#### 不要主提

- 命名、常量、日志等通用规范问题
- 纯架构边界和职责归属问题
- 索引与事务细节
- 压测级性能问题
- 测试覆盖率是否充足

#### 一句话

它负责回答：

**这段代码的行为会不会错。**

---

### 6.4 可维护性与代码健康专家

#### 核心职责

识别那些当前能跑，但已经明显提高后续维护、调试、测试和演化成本的实现。

#### 主提范围

- 函数过长
- 分支过深
- 复制粘贴
- 职责揉在一起
- 抽象不清
- 结构难懂
- 测试和调试成本显著上升

#### 不要主提

- 业务规则错误
- 边界职责错误
- 命名和常量等通用规范问题
- SQL / 索引 / 事务
- 系统级性能和安全问题

#### 一句话

它负责回答：

**这段代码后面好不好维护。**

---

### 6.5 数据库分析专家

#### 核心职责

判断数据库内部语义、事务边界、schema 兼容性和索引设计是否有风险。

#### 主提范围

- SQL 语义错误
- 事务、锁、回滚边界不完整
- Schema 变更兼容性问题
- 索引、默认值、约束不合理
- 迁移和数据模型演进风险

#### 不要主提

- 命名和代码风格
- 通用架构问题
- 系统级故障恢复和容量问题

#### 一句话

它负责回答：

**数据库内部边界和语义有没有问题。**

---

### 6.6 性能与可靠性专家

#### 核心职责

判断代码上线后是否稳定，是否会放大系统压力。

#### 主提范围

- 热点路径退化
- 批处理过大
- 同步阻塞
- 并发争用
- 锁放大
- 超时、重试、降级、回滚、故障恢复缺口
- 局部故障放大成系统性压力

#### 不要主提

- 具体索引设计
- SQL 字段问题
- 命名和通用风格问题

#### 一句话

它负责回答：

**这段代码上线后稳不稳。**

---

### 6.7 Redis 分析专家

#### 核心职责

判断缓存链路是否会脏、炸、失控。

#### 主提范围

- 缓存一致性
- TTL 与失效策略
- 热点 key
- Lua / 事务 / 多 key 原子性

#### 一句话

它负责回答：

**缓存链路会不会出问题。**

---

### 6.8 MQ 分析专家

#### 核心职责

判断消息链路是否会乱、丢、重。

#### 主提范围

- 幂等与去重
- 顺序消费
- ack 与超时
- 重试、补偿、死信
- 堆积治理

#### 一句话

它负责回答：

**消息链路是否可靠。**

---

### 6.9 安全与合规专家

#### 核心职责

判断代码是否突破安全边界。

#### 主提范围

- 鉴权授权
- 输入校验绕过
- 敏感数据处理
- 合规要求

#### 一句话

它负责回答：

**这段代码会不会破安全边界。**

---

### 6.10 测试与验证专家

#### 核心职责

判断当前改动有没有足够的保护网来防止回归。

#### 主提范围

- 缺测试
- 断言太弱
- 高风险路径无保护
- 回归保护不足
- 缺集成测试或人工验证步骤

#### 一句话

它负责回答：

**这次改动有没有足够保护网。**

---

### 6.11 前端可访问性专家

#### 核心职责

先收窄为前端 a11y 与关键可达性回归。

#### 主提范围

- a11y 回归
- 关键交互不可达
- 基础渲染与可达性问题

#### 一句话

它负责回答：

**关键前端交互对用户是否还可达。**

## 8. 路由原则

路由不要只靠目录名或文件名，应该更多依赖“问题信号”。

### 7.1 通用编码规范专家信号

- 魔法值
- catch 后吞异常
- 日志缺关键字段
- 布尔命名差
- 常量和枚举使用混乱
- 判空顺序危险
- 集合和泛型使用不规范
- 同类代码风格明显不一致

### 7.2 DDD 架构专家信号

- domain / aggregate / entity / value object
- application / service / repository 跨层交叉
- 聚合边界变化
- 应用服务承担业务规则
- 基础设施实现上浮
- 跨模块直接依赖

### 7.3 正确性与业务专家信号

- comment / 注释 / TODO / FIXME / 未实现
- 状态流转判断
- 输入输出和副作用路径
- 关键业务条件
- 异常与边界输入分支

### 7.4 可维护性专家信号

- 超长函数
- 深层嵌套
- 大段复制粘贴
- 一个对象承担太多职责
- 结构复杂但行为暂时没错

---

## 9. merge 归因规则

当多个专家都提了时，最终结果层按下面顺序收口：

### 规则 1：行为错误优先

如果问题同时满足：

- 注释没实现
- 命名也不好
- 结构也一般

最终优先归到：

- `correctness_business`

### 规则 2：边界错误优先于写法问题

如果一段代码同时存在：

- 依赖方向错误
- 命名不清

最终优先归到：

- `ddd_architecture`

### 规则 3：写法问题优先于纯维护建议

如果没有行为错误，也没有边界错误，但：

- 魔法值很多
- 日志不规范
- 判空写法危险

最终优先归到：

- 通用编码规范专家

只有当它不属于具体规范问题，而是整体结构后续维护代价高，才归可维护性专家。

---

## 10. 典型问题归因示例

| 场景 | 应归专家 | 原因 |
|---|---|---|
| 注释写着“失败自动降级”，代码里没有降级逻辑 | `correctness_business` | 行为承诺未落地 |
| service 里直接 new repository implementation 并夹带业务规则 | `ddd_architecture` | 职责和边界错位 |
| 方法里大量硬编码字符串状态值 | 通用编码规范专家 | 典型常规 Java 规范问题 |
| 一个函数 200 行，分支很多，复制粘贴多，但行为暂时没错 | `maintainability_code_health` | 主要风险是长期演化成本 |
| catch 后只打日志不处理，且日志里没有关键主键 | 通用编码规范专家 | 异常与日志使用不规范 |
| 输入校验没做导致恶意参数可绕过校验 | `security_compliance` | 核心是安全边界 |
| SQL 删除了 LIMIT，批量读取边界消失 | `database_analysis` | 核心是数据库语义和数据边界 |
| 批处理过大导致锁持有时间过长，系统压力被放大 | `performance_reliability` | 核心是系统级稳定性 |

## 11. 后续落地顺序

建议按下面顺序落，风险最小。

### 第一步：先改 prompt 边界文案

先把以下几个专家的“负责什么 / 不负责什么”写清楚：

- 通用编码规范专家
- DDD 架构专家
- 正确性与业务专家
- 可维护性与代码健康专家

### 第二步：再改 merge 归因逻辑

按第 8 节规则做唯一归因。

### 第三步：最后再调专家路由

等前两步稳定，再调 activation hints 和主 Agent 路由。

## 12. 最终建议结构

如果按这份手册走，关键专家会变成下面这组稳定结构：

1. 通用编码规范专家  
   看普通 Java 代码写法是不是规整

2. DDD 架构专家  
   看职责、边界和依赖方向是不是放对

3. 正确性与业务专家  
   看行为和结果会不会错

4. 可维护性与代码健康专家  
   看后续维护和演化成本

5. 数据库 / Redis / MQ / 性能 / 安全 / 测试 / 前端可访问性专家  
   继续承担各自专项问题

一句话总结：

**这份手册的目标不是让专家更多，而是让每类问题都有唯一主责，不再重复提。**
