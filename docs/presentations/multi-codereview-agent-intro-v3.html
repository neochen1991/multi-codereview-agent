Experts 03 / 06

10+ 个领域专家，按问题责任边界分工

这里不是为了把角色堆多，而是让不同类型的问题由对应领域来负责。复杂 MR 里的问题，本来就不该用同一种视角去看。

专家划分的原则，不是按技术名词分，而是按“什么问题该由谁负责”来分。

业务与架构

主要回答：逻辑对不对，设计有没有退化。

业务正确性专家 看状态流转、边界条件、副作用。
架构与设计专家 看聚合边界、依赖方向、职责划分。
可维护性专家 看复杂度、重复逻辑、表达清晰度。

基础设施

主要回答：技术实现稳不稳、扛不扛得住。

数据库专家 看事务、索引、查询性能、批量操作。
缓存 / Redis 专家 看缓存一致性、热点放大、失效策略。
MQ / 性能专家 看消息可靠性、容量风险、资源使用。

质量保障

主要回答：风险有没有被验证住、兜住。

测试覆盖专家 看关键路径有没有测试保护。
安全合规专家 看权限边界、注入风险、敏感信息。
前端体验专家 看可访问性、交互一致性和页面风险。
一个典型复杂 MR，往往会同时拉起业务正确性、架构、数据库和性能几个专家分别审查，最后再统一去重和收敛。
Multi Agent 04 / 06

为什么复杂代码审查需要多 Agent 协作？

多 Agent 的核心价值，不只是管理上下文窗口，而是把复杂代码审查从“一个模型泛泛地看”变成“多领域分工协作地看”。

单 Agent 的问题

  • 所有问题混在一个上下文里：业务、架构、数据库、性能问题会互相稀释，容易什么都提一点、但都不够深。
  • 不同领域互相干扰：模型容易用同一套判断方式去看完全不同类型的问题。
  • 输出难以收敛：重复、噪音和真正重要的问题容易混在一起。
  • 上下文窗口不是唯一问题：即使窗口足够大，专业分工不清的问题依然存在。

多 Agent 的优势

  • 专业分工：每个 Agent 只负责一个领域，在自己的职责边界内深看。
  • 证据聚焦：每个专家只拿相关文档、规则、代码和外部证据。
  • 并行执行：一个专家慢一点，不会拖住所有领域一起停下来。
  • 结果收敛：最后通过 judge、阈值和治理逻辑，留下真正值得优先处理的问题。
多 Agent 的核心价值，不是“把大任务拆小来适应上下文窗口”，而是让复杂代码审查变成一条有专业分工、证据隔离和结果收敛的可控链路。
Framework 05 / 06

为什么选择 LangGraph 作为流程编排框架？

我们面对的不是简单对话，而是一条需要状态管理、流程编排和治理能力的复杂审查链路。选型时，我们关心的不是“谁更容易起步”，而是“谁更适合把这条链路做成工程系统”。

我们真正要跑的是这样一条链
选专家 决定谁参与
补证据 拉规则、代码、外部信息
并行审查 多个专家分别判断
结果收敛 去重、过滤、排序
人工介入 必要时暂停和确认
我们真正需要什么

把专家选择、补证据、并行审查、结果收敛放进一条可执行链路里。

直接调模型 API

适合起步快，但流程和状态都要自己在外面补。

AutoGen / CrewAI

更适合多角色协作对话，但流程治理和状态管理需要额外建设。

LangGraph

天然适合多节点、多状态、多阶段的流程编排。

我们特别看重什么

状态管理、失败恢复、人工门禁、过程回放和治理能力。

直接调模型 API 的问题

流程一复杂，脚本会越来越重，后面很难治理。

多 Agent 对话框架的问题

适合“大家一起聊”，不一定适合“把流程图跑稳”。

LangGraph 的优势

图就是流程，状态就是核心，后面做治理和回放都更顺。

一句话总结 选择 LangGraph，不是因为它更热，而是因为它最适合把这条复杂流程链组织清楚、管理清楚、回放清楚。
