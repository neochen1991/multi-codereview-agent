Architecture

总体架构

架构设计遵循“入口、编排、执行、证据”四层拆分。其目标不是提升模型自由度，而是使监控入口、执行策略、 专家角色与外部依赖之间形成明确边界，从而为稳定性治理和能力扩展提供支撑。

Execution Discipline

专家执行链路

当前实现并未将专家执行视为开放式对话，而是以“命令先行、上下文受控、输出结构化、结果可审计”为基本约束。 这使系统能够将多 Agent 协作纳入统一的治理框架。

1

命令先行

先由主 Agent 发出可审计命令，再由专家执行，防止执行链路失控。

2

上下文受控

Tool、Skill、MCP 仅在命令门禁允许时注入，避免上下文无边界膨胀。

3

输出结构化

关键专家输出需可解析、可回写、可进入最终裁决，而非停留在自然语言散文层面。

Operational Loop

从页面巡检到自动 RCA

Step 01

巡检配置

在“红蓝对抗感知”页面配置目标页面、Cookie、巡检周期与感知策略。

Step 02

异常捕获

通过 Playwright 访问页面，识别控制台错误、接口失败和关键交互异常，必要时降级为 HTTP 巡检。

Step 03

事件入库

监控事件与异常上下文写入历史记录，并自动生成 incident 与调查任务。

Step 04

分析收口

系统按模式和预算调度专家轮次，由 Judge 汇总证据并形成最终报告。

设计意义

将分析建立在可验证证据之上

当前体系已将“能力扩展”与“证据采集”从 Prompt 中抽离，转化为可配置、可绑定、可审计的工程能力面， 这为后续接入更多生产系统奠定了基础。

Governance

治理约束

01

命令先行

主 Agent 必须先下发命令，专家执行不得绕过主控链路。

02

有效结论门禁

在严格模式下，最终结论不得以“继续分析”替代根因判断。

03

会话终态保障

任何异常任务均需进入终态并记录原因，避免长期悬挂。

04

断点续写

中断后优先依赖 checkpoint 恢复，降低长流程重跑成本。

05

轨迹优先

命令、工具、证据、反驳与裁决均需可回放，而非仅保留最终结论。

06

系统与模型分层

模型负责推理，系统负责状态机、超时、重试、审计与降级。

当前风险

下一阶段的关键在于稳定性，而非能力展示

模型超时管理、quick 与 standard 模式分层、专家完成率、证据置信度计算以及自动化处置闭环， 是后续工程深化的主要关注点。

Recommendation

建议的后续投入方向

1

继续强化运行稳定性

围绕弱并发模型优化超时、降级与裁决策略，并进一步稳定 quick / standard 模式的边界。

2

扩展自动化运维闭环

从自动感知与自动分析延伸至案例检索、处置建议生成和受控动作执行。

3

建立标准评测体系

通过真实故障案例评测专家完成率、证据准确率和报告可用性，形成持续演进的量化基线。
