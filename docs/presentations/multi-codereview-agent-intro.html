Multi-agent Code Review

复杂 MR 的代码审查，正在变成团队的稳定性问题

这个项目的目标，不是再做一个会提意见的 AI 工具，而是把复杂代码审查沉淀成一套团队可以稳定复用的能力。

减少高价值问题漏审
降低对个人经验的依赖
让审查意见真正进入研发链路
项目简介 01 / 11
问题越来越复合 业务、架构、数据库、消息和性能问题经常交织在一起，单一视角很难稳定看透。
经验过度依赖个人 很多高价值问题，今天仍然主要靠少数资深同学的 review 经验来兜底。
多专家协同是解法 系统先选专家再分工审查，让不同领域问题由对应角色分别处理，而不是一个模型硬看全部。
结果直接回到 CodeHub 筛选后的有效问题可以直接回写到 CodeHub MR，进入研发本来的协作闭环。
下面这份介绍重点回答 7 个问题：为什么现在要做、系统怎样支撑、为什么一定要多专家协同，以及它现在到底做到什么程度。
Value 02 / 11

这套工具，直接改善了 Code Review 的四件事

如果只从 Code Review 本身来看，它的价值不在于“多提一些意见”，而在于把复杂问题看得更全、把结论说得更准、把后续处理成本降下来。

把复杂风险看得更全

一个 MR 不再只看语法、空指针和命名，而是会同时去看业务正确性、架构边界、事务顺序、数据库和性能风险。

把“为什么有问题”讲清楚

它不只是给一句泛泛的提醒，而是尽量把命中的规则、相关代码、补充证据和修改建议一起给出来，让 reviewer 更容易判断。

让意见更容易落地

经过阈值过滤后的有效问题，可以直接回写到 MR 讨论里，研发看到后就能顺着原始代码位置和证据去改，不需要再二次转述。

对 reviewer 的价值

相当于先帮 reviewer 做了一轮分领域预审，把真正值得花时间确认的问题先捞出来。

对研发的价值

拿到的不是“这里可能有点问题”，而是带上下文、带证据、带修改方向的检视结论，处理成本更低。

对团队质量的价值

资深同学平时 review 的关注点，可以逐步沉淀成规则、文档和工具，不再只存在于少数人的经验里。

Architecture 03 / 11

这套架构的核心，不是堆能力，而是把审查链路拆清楚

可以把它理解成一栋五层房子：每一层都只解决一类问题，入口、编排、分析、扩展和支撑各自分工。

用户入口层 · 首页 / 审核工作台 / 历史记录 / 设置

左侧墙体

  • 解决协作入口问题
  • 承接项目配置与知识管理
  • 统一任务发起与治理入口

应用服务层

ReviewService 负责把任务创建、排队、状态流转和结果输出串成一条稳定流程。
RuntimeSettings + PlatformAdapter 负责把项目配置、代码仓和不同 Git 平台输入整理成统一审查对象。

智能分析层

Main Agent + Expert Agents 负责专家选择、任务分发、分领域审查和最终问题收敛。
Rules / Skills / Tools 负责补规则、补代码、补知识和补外部证据，让审查不是只看 diff。

右侧墙体

  • 承接有效问题闭环
  • 支撑过程与治理观测

扩展层

平台支撑层

代码与数据源

Implementation 04 / 11

一次 MR 审核，系统实际经历了什么

这不是一次简单的大模型调用，而是一条有组织的审查流程：先接入代码，再选专家、补证据、执行审查，最后收敛有效问题。

1. 接入代码变更

从 Git 平台拿到 MR 信息、变更文件和 diff，统一成可审查输入。

2. 选择参与专家

主 Agent 判断这次应该由哪些专家参与，并完成派工分发。

3. 补充审查证据

补源码上下文、规则命中、知识文档、设计文档和外部数据，不让专家只靠 diff 猜。

4. 分领域执行审查

各专家在自己的职责范围内给出发现、证据和修改建议。

5. 收敛有效问题

通过阈值、judge 和治理逻辑，把真正值得处理的问题留下来。

不是一次模型直出
每一步都有明确职责
结果最终进入治理闭环
Experts 05 / 11

专家划分的原则，是按问题责任边界分工

这里不是为了“专家看起来很多”，而是为了让不同类型的问题由对应领域来负责，这样既不会重复，也不会漏看。

专家划分的原则，不是按技术名词罗列角色，而是按“什么问题该由谁负责”来切分责任边界。
第一组：业务和架构

正确性与业务、架构与设计、DDD、可维护性，主要看规则有没有改坏、边界有没有乱、以后会不会越来越难维护。

第二组：基础设施

数据库、Redis、MQ、性能与可靠性，主要看事务、缓存、一致性、消息和容量风险。

第三组：质量保障

测试与验证、安全与合规、前端可访问性，主要看测试保护、权限边界和前端体验。

架构与设计专家

重点看边界、依赖和抽象有没有被改乱，判断设计是不是在退化。

正确性与业务专家

重点看状态流转、边界条件和副作用，判断业务规则是不是被改坏了。

数据库 / MQ / Redis / 性能专家

重点看事务、索引、消息顺序、缓存一致性、批处理和资源稳不稳。

Extensibility 06 / 11

为什么这套系统可以持续扩展，而不是一次性工程

主链路只负责编排，领域知识和外部能力都挂在专家侧，所以新能力可以持续往上加，而不是每次都回头重改主系统。

平时怎么扩

系统把扩展点拆成三层，不同层只做自己那一件事，所以后面加能力时不用总去改主流程。

规范文档 回答“这个专家该按什么规则看”。文档上传后会拆章节、拆规则卡。
↓
Skill 回答“什么场景该启什么能力”。用 SKILL.md 描述触发条件和依赖。
↓
Tool 负责真正去拿证据。比如搜代码、查知识、读设计文档、拉 PG 元信息。

运行时怎么组上下文

专家真正开跑前，不会把所有文档和代码全文都塞进去，而是先看这次 MR 到底和哪些东西有关。

MR 输入 变更文件、diff、主 Agent 路由理由
规则命中 先预筛，只挑这轮值得启用的规则
代码证据 目标 hunk、repo 命中源码片段、关联上下文
外部证据 知识库、设计文档、数据库元信息、运行时工具结果
最后交给专家的，不是一大包杂糅信息

而是“这轮最相关的一小包上下文”。这样更聚焦，也更像一个专业 reviewer 在看自己该看的东西。

为什么这种设计更值钱

第一，扩展点开放得很明确，后面要加新能力时，优先加在专家中心和 extensions/ 就行。

为什么比“全塞 prompt”更稳

因为上下文先经过筛选和裁剪，不会一上来就把几千行规范、整仓代码和一堆噪音一起塞给模型。

为什么后面更好治理

每一步都能回放，能看到哪个规则命中了、启了哪个 skill、tool 拉回了什么证据，而不是黑盒出结论。

Multi Agent 07 / 11

为什么复杂代码审查不能只靠单 Agent

多 Agent 的核心价值，不只是上下文窗口管理，而是专业分工、证据隔离和结果收敛。

单 Agent 的天然问题

  • 所有问题混在一个上下文里，复杂问题容易互相稀释。
  • 业务、架构、数据库、性能问题很容易被平均化处理。
  • 输出看起来什么都覆盖了，但深度和稳定性不够一致。
  • 一旦输入太杂，模型就更容易给出泛化而不聚焦的结论。

多 Agent 的核心收益

  • 每类问题由对应专家处理，分工边界清楚。
  • 每个专家只拿和自己职责相关的上下文，证据更聚焦。
  • 不同领域可以并行补证据，不必互相拖累。
  • 最后还能由 judge、阈值和治理逻辑统一收敛结果。
多 Agent 的真正价值，不只是上下文窗口管理，而是把复杂代码审查拆成“专业分工、证据隔离、结果收敛”三件事分别做好。
Framework 08 / 11

为什么最终选择 LangGraph

我们面对的是一条复杂审查流程，而不是一个简单对话 Agent。选型的关键，是它能不能把流程、状态和治理真正管起来。

我们真正需要什么

多专家协作、状态流转、失败重试、人工门禁、过程回放和治理观测，都要放进同一条链路里。

对其他框架的判断

对话式 Agent、轻量工具编排和快速原型没有问题，但一旦流程变复杂，表达和治理成本会明显上来。

为什么最后选 LangGraph

它最适合把审查流程画成一张真正可执行、可中断、可回放的图，而不是一串越来越长的 chain。

流程编排

要把专家选择、补证据、judge 和阈值治理放在一条清楚的主链路里。

其他框架的常见问题

更适合“多个角色一起说话”，但对多分支流程的表达通常不够直观。

LangGraph 的优势

节点和边就是流程本身，天然更适合多阶段、多分支的执行逻辑。

状态管理

每次审查都需要保留专家选择、运行证据、issue 收敛和人工介入状态。

其他框架的常见问题

状态可以补，但往往不是第一公民，流程一复杂就容易变散。

LangGraph 的优势

状态是主结构的一部分，更适合长流程、长任务和需要恢复的执行场景。

治理与回放

我们需要知道专家为什么参与、用了什么规则、调了哪些工具、最后为什么收敛成这个结果。

其他框架的常见问题

能做执行，但要补成“可治理、可回放”的工程体系，代价通常更高。

LangGraph 的优势

很适合把复杂审查流程做成一条可观测、可治理的工程链，而不是黑盒对话。

我们最终选择 LangGraph，不是因为它更流行，而是因为这类多专家代码审查问题，本质上就是一条需要状态管理和治理能力的复杂流程。

Example 10 / 11

一个真实案例：它抓到的不是表面问题，而是高价值风险

这个案例的关键，不是命中了多少条问题，而是命中的问题本身足够复杂，而且证据链是完整的。

PR 改了什么

真实 Java DDD 场景下，对聚合创建、事件发布顺序和批量读取链路进行了改动，影响面不是单一模块。

系统命中了什么
  • aggregate factory bypass
  • 事件发布与持久化顺序倒置
  • 删除 LIMIT / 批量读取边界缺失
哪些专家给了证据
  • DDD / 架构专家：看边界和职责
  • 正确性专家：看时序和副作用
  • 数据库 / 性能专家：看批量读取与大结果集风险
为什么这个案例值钱

这些问题都不是表面代码味道，而是人工 review 也容易漏掉、但一旦漏掉代价会更高的工程风险。

这个案例真正说明的是：系统已经不只是“看 diff 提几个通用意见”，而是能围绕真实证据，把高价值复杂问题稳定带到结果层。
